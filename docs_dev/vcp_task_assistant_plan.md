# VCPTaskAssistant 改造计划

## 目标

将 VCPTaskAssistant 从“任务派发中心”升级为“有回声、有验收、有记录的任务执行中心”。

本次改造不追求把 Agent 变成机械工单执行器，而是在保留 VCP 伙伴式、可成长性的设计哲学前提下，补齐任务系统的完成闭环：

- 任务必须有明确回声。
- 系统必须有客观验收。
- 管理面板必须能看到真实执行记录。
- “派发成功”不能再显示为“任务完成”。

## 当前共识

| 事项 | 结论 |
|------|------|
| 成功判定 | 不能只看 HTTP 2xx，必须解析插件返回体 |
| 运行记录 | 需要独立 run 文件，保存真实回执和错误 |
| 任务模式 | 不保留单纯派发成功语义，任务默认需要完成回执 |
| 异步委托 | 任务应优先走 AgentAssistant 异步委托闭环 |
| 产出验收 | 可配置验收规则，由系统检查真实产物 |
| 前端状态 | 全部中文化，不再展示 `success (1 agents)` |
| 回执协议 | 使用有 VCP 气质的“任务回声”，避免僵硬工程交付口吻 |

## 阶段 1：现有结构梳理

### 涉及文件

| 文件 | 作用 |
|------|------|
| `Plugin/VCPTaskAssistant/vcp-task-assistant.js` | 任务配置、调度、触发、历史记录 |
| `Plugin/VCPTaskAssistant/task-center-data.json` | 当前任务和历史记录持久化 |
| `routes/admin/taskAssistant.js` | 管理面板任务 API |
| `Plugin/AgentAssistant/AgentAssistant.js` | Agent 通讯与异步委托 |
| `server.js` | `/v1/human/tool` 直接工具调用入口 |

### 已确认问题

- `/v1/human/tool` 会用 HTTP 200 返回插件执行结果对象，因此 HTTP 200 不等于插件成功。
- `VCPTaskAssistant` 当前只记录派发结果，没有保存 Agent 的真实回复。
- 当前 `history` 中的 `success` 实际是“至少一个 Agent 收到请求”，不是“任务完成”。
- 当前任务提示词没有系统级完成回声契约。
- 当前没有产物验收机制。

## 阶段 2：中文状态机设计

### 新状态

| 状态 | 含义 |
|------|------|
| 等待执行 | 任务已配置，尚未到执行时间 |
| 执行中 | 任务已开始运行 |
| 已提交委托 | AgentAssistant 已创建异步委托，等待最终回声 |
| 等待回声 | 已唤醒 Agent，但尚未拿到可解析任务回声 |
| 已完成 | Agent 回声为已完成，且系统验收通过 |
| 执行失败 | Agent 明确失败，或插件链路返回错误 |
| 等待超时 | 超过任务配置等待时间 |
| 验收未通过 | Agent 声称完成，但系统未验收到产物 |
| 旧记录未验证 | 历史记录兼容状态 |

### 旧状态映射

| 旧值 | 新显示 |
|------|--------|
| `success (...)` | 旧记录未验证 |
| `partial_success (...)` | 旧记录未验证 |
| `error: ...` | 执行失败 |
| `running via ...` | 执行中 |

## 阶段 3：任务回声协议

### 设计原则

- 回声是 Agent 对自身行动的自述，不是冷冰冰的工单报告。
- 系统可以解析固定标题和字段，但字段文字保持中文、自然、伙伴感。
- Agent 可以诚实说未完成、需要等待、需要彦确认。
- 回声不替代系统验收；回声是主观完成声明，验收是客观产物检查。

### 建议协议

```text
【任务回声】
状态：已完成 / 未完成 / 需要继续等待 / 需要彦确认

本轮我实际做了：
- ...

我留下的痕迹：
- 写入了：...
- 创建了：...
- 更新了：...
- 没有产生文件，但完成了：...

我确认完成的依据：
- ...

仍然悬而未决的部分：
- 无 / ...

给彦的简短说明：
...
【任务回声结束】
```

### 失败回声

```text
【任务回声】
状态：未完成

我卡住的位置：
...

原因：
...

我已经尝试过：
...

需要彦确认或补充的是：
...
【任务回声结束】
```

### 等待回声

```text
【任务回声】
状态：需要继续等待

当前进展：
...

下一次醒来建议：
...

建议等待：10分钟
【任务回声结束】
```

### 自动包裹的任务中心提示

```text
你正在被 VCPTaskAssistant 温和唤醒，处理一项约定任务。
请认真完成任务本身，而不是只复述任务说明。
如果你留下了日记、文件、话题或其他痕迹，请在最后用【任务回声】说明。
如果你无法完成，也请诚实说明卡住的位置。
```

## 阶段 4：系统验收规则

### 配置概念

面向用户和前端展示使用中文概念：

```json
{
  "验收规则": [
    {
      "类型": "日记新增",
      "日记本": "大师的记忆整理",
      "开始时间": "任务开始后",
      "必须包含": "记忆大师V2 梦系统整理线索登记"
    }
  ]
}
```

代码内部可以使用英文变量名，但管理面板、任务记录、错误描述尽量中文化。

### 首批验收类型

| 类型 | 检查方式 |
|------|----------|
| 日记新增 | 检查指定日记本在任务开始后是否新增文件 |
| 日记包含 | 检查新增文件中是否包含指定关键词 |
| 文件新增 | 检查指定目录是否新增文件 |
| 回声包含 | 检查任务回声是否包含指定内容 |

### 验收结果

| 结果 | 状态 |
|------|------|
| 回声已完成，验收通过 | 已完成 |
| 回声已完成，验收失败 | 验收未通过 |
| 回声未完成 | 执行失败 |
| 无验收规则但有完成回声 | 已完成，标记“未配置验收规则” |
| 无回声 | 等待超时或等待回声 |

## 阶段 5：后端执行链路调整

### 运行记录独立化

新增目录：

```text
Plugin/VCPTaskAssistant/task-runs/
```

每次运行写入：

```text
Plugin/VCPTaskAssistant/task-runs/{runId}.json
```

### run 文件建议结构

```json
{
  "运行ID": "run_...",
  "任务ID": "...",
  "任务名称": "...",
  "触发来源": "手动触发",
  "开始时间": "...",
  "结束时间": "...",
  "状态": "已完成",
  "目标Agent": ["曦月"],
  "委托ID": "aa-delegation-...",
  "任务提示词摘要": "...",
  "任务回声": {
    "状态": "已完成",
    "原文": "...",
    "解析结果": {}
  },
  "系统验收": [
    {
      "类型": "日记新增",
      "结果": "通过",
      "证据": ["dailynote/...txt"]
    }
  ],
  "错误": null
}
```

### 主数据文件保留摘要

`task-center-data.json` 只保存：

- 当前任务配置
- 最近运行摘要
- `runId`
- 中文状态
- 最近错误摘要
- 最近产物路径

### 执行链路建议

1. 创建 `runId`，写入 `执行中`。
2. 自动包裹任务中心提示。
3. 调用 AgentAssistant，优先使用异步委托。
4. 获取委托 ID 后记录 `已提交委托`。
5. 后台轮询或接收回调，等待任务回声。
6. 解析任务回声。
7. 执行系统验收规则。
8. 写入最终状态和完整 run 文件。
9. 管理面板读取 run 摘要和详情。

## 阶段 7：旧数据兼容

### 原则

- 不改写旧历史事实。
- 不把旧 `success` 自动认定为已完成。
- 前端展示为“旧记录未验证”。

### 迁移策略

- 读取旧 history 时动态映射显示。
- 新 run 使用新结构。
- 可选提供一次性迁移脚本，把旧 history 生成只读 run 摘要。

## 阶段 8：验证场景

### 场景 A：共鸣心跳

目标：

- 手动触发后，曦月与律曦均返回任务回声。
- 至少一个 Agent 在 `[公共的日常]` 留下本轮心跳记录。
- 任务中心显示真实产物路径。

验收：

- 状态为“已完成”。
- run 详情中能看到两个 Agent 的回声。
- 验收规则能定位新增日记。

### 场景 B：记忆大师梦系统整理线索巡检

目标：

- 大师扫描 `[曦月的梦]`。
- 若发现线索，写入 `[大师的记忆整理]`。
- 若无新增线索，也返回清楚的任务回声。

验收：

- 有新增线索时，检查 `[大师的记忆整理]` 新文件。
- 无新增线索时，回声说明“本轮无新增线索”，状态仍可为“已完成”。

### 场景 C：故障验证

目标：

- 模拟 AgentAssistant 返回 `{ status: "error" }`。
- 模拟模型上下文超限。
- 模拟 Agent 声称完成但未产生日记。

验收：

- 插件错误显示“执行失败”。
- 超时显示“等待超时”或“执行失败”。
- 未产生产物显示“验收未通过”。

## 决策记录

| 决策 | 理由 | 日期 |
|------|------|------|
| 不保留单纯派发成功语义 | 对任务中心来说，派发成功没有用户价值 | 2026-05-02 |
| 使用“任务回声”而非“完成报告” | 更符合 VCP 伙伴式 Agent 设计哲学 | 2026-05-02 |
| run 明细独立落盘 | 避免 `task-center-data.json` 膨胀，同时提升可追溯性 | 2026-05-02 |
| 旧 success 显示为旧记录未验证 | 避免历史记录造成假完成感 | 2026-05-02 |

## 风险与注意事项

| 风险 | 说明 | 缓解 |
|------|------|------|
| 任务回声解析失败 | Agent 可能未按格式回复 | 自动补问一次，仍失败则标记等待回声或失败 |
| 异步委托时间过长 | 管理接口不能长时间阻塞 | 手动触发只等待提交成功，后台更新最终状态 |
| 断言过强导致误判 | 有些任务没有文件产物 | 允许配置无产物任务，但必须有任务回声 |
| 中文字段影响代码维护 | 内部逻辑可用英文变量名 | 对外展示和配置说明中文化 |
| 旧数据兼容复杂 | 旧 history 缺少真实证据 | 显示为旧记录未验证，不做强迁移 |

## 推荐实施顺序

1. 改成功判定：解析 `/v1/human/tool` 返回体。
2. 增加 `task-runs/{runId}.json`，保存完整运行证据。
3. 引入中文状态机。
4. 默认启用任务回声包裹提示。
5. 接入 AgentAssistant 异步委托与回调/轮询。
6. 增加首批验收规则。
7. 调整管理面板显示。
8. 用真实两条任务做回归验证。

## 补充设计：运行记录进入现有面板

### 结论

run 详情应直接集成到现有“任务派发中心”面板，而不是另开一个独立页面。

当前面板已经有：

- 任务当前状态，例如“待机”“执行中”。
- 最近结果。
- 成功/失败/总计统计。
- 最近执行记录列表。

因此改造后可以在原位置升级显示：

- 当前状态：从 `待机/执行中` 扩展为中文状态机。
- 最近结果：从 `success (2 agents)` 改为 `已完成 / 验收未通过 / 执行失败 / 旧记录未验证`。
- 最近执行记录：每条记录增加“查看回声”“查看验收”“查看产物”入口。
- 详情抽屉或弹窗：展示 run 文件中的完整执行证据。

### 面板交互建议

| 区域 | 调整 |
|------|------|
| 当前状态卡 | 显示中文状态、最近 runId、委托 ID、等待时长 |
| 最近结果 | 显示人类可读状态，不展示英文内部状态 |
| 最近执行记录 | 每条记录右侧增加同风格下拉/展开按钮，展开后查看任务回声、系统验收、产物和错误详情 |
| 任务编辑区 | 增加“验收规则”配置块 |
| 手动执行按钮 | 点击后立即显示“已提交委托”或“执行中”，后台刷新最终状态 |

### 后端 API 建议

保留现有接口，同时增加：

| 接口 | 用途 |
|------|------|
| `GET /task-assistant/runs` | 获取最近运行记录摘要 |
| `GET /task-assistant/runs/:runId` | 获取单次运行详情 |
| `GET /task-assistant/tasks/:taskId/runs` | 获取指定任务的运行记录 |

现有 `GET /task-assistant/status` 继续返回任务当前状态和最近摘要，避免前端每次都读取完整 run 文件。

## 补充设计：任务配置拆分

### 结论

`task-center-data.json` 可以拆分，而且建议拆分。

当前所有任务、运行态、历史记录都放在一个文件里，短期简单，长期会有几个问题：

- 任一任务保存都会重写整个文件。
- run 历史增长会让配置文件膨胀。
- 单个任务损坏可能影响全部任务加载。
- 不方便人工备份、迁移、对比某一个任务。
- 不方便以后为每个任务附加验收规则、运行摘要、权限边界。

### 推荐目录结构

```text
Plugin/VCPTaskAssistant/
|- task-center-data.json          # 全局设置与兼容索引
|- tasks/
|  |- task_xxx.json               # 单个任务配置
|  `- task_yyy.json
|- task-runs/
|  |- run_xxx.json                # 单次运行详情
|  `- run_yyy.json
`- task-runtime/
   |- task_xxx.runtime.json       # 可选：单任务运行态摘要
   `- task_yyy.runtime.json
```

### 文件职责

| 文件 | 职责 |
|------|------|
| `task-center-data.json` | 只保存全局开关、设置、版本、任务索引、兼容信息 |
| `tasks/{taskId}.json` | 保存任务定义、提示词模板、目标 Agent、调度、验收规则 |
| `task-runtime/{taskId}.runtime.json` | 保存该任务当前状态、最近 runId、统计计数、下次执行时间 |
| `task-runs/{runId}.json` | 保存单次运行详情、任务回声、系统验收、错误、产物 |

### 更简化的首版方案

为了减少第一轮改造量，也可以先做三层：

```text
task-center-data.json       # 全局设置 + 任务索引 + 旧数据兼容
tasks/{taskId}.json         # 任务配置 + runtime 摘要
task-runs/{runId}.json      # 运行详情
```

首版不单独拆 `task-runtime/`，等运行态字段变多后再拆。

### 加载流程

1. 启动时读取 `task-center-data.json`。
2. 如果发现旧结构里有 `tasks` 数组，则执行内存级兼容加载。
3. 优先读取 `tasks/*.json` 作为真实任务配置。
4. 根据任务配置重建调度器。
5. 保存任务时只写对应 `tasks/{taskId}.json`。
6. 更新运行态时只写对应任务文件或 runtime 文件。

### 迁移策略

- 首次启动如果发现旧 `task-center-data.json.tasks`，自动生成 `tasks/{taskId}.json`。
- 原文件保留备份，例如 `task-center-data.json.bak-before-split-时间戳`。
- 迁移后 `task-center-data.json` 只保留全局字段和任务索引。
- 旧 history 显示为“旧记录未验证”，不强行生成完成证据。

### 注意事项

- 对外展示可以使用中文字段概念，代码内部仍可映射成英文变量，避免维护困难。
- 需要文件写入原子化：先写临时文件，再替换正式文件，避免半写入损坏任务。
- 任务 ID 应保持稳定，不能因为任务改名而变化。
- 删除任务时不要删除历史 run，改为任务配置删除、run 保留归档。

### 结论

后续对 VCP 后端“任务派发中心”管理面板的改动，必须延续现有页面风格，不另起一套 UI 语言。

当前 `AdminPanel-Vue/src/views/ForumAssistantConfig.vue` 已经形成了比较明确的视觉结构：

- 外层仍使用 `config-section active-section forum-assistant-view`。
- 主要内容仍使用 `card`、`status-card`、`composer-card`、`history-card`。
- 单个任务仍使用 `task-card`，运行态仍放在 `runtime-panel`。
- 状态标签仍使用 `status-badge`，只扩展中文状态映射与颜色语义。
- 表单仍使用现有 `field`、`switch-row`、`task-grid`。
- 历史记录仍使用 `history-list`、`history-item`、`history-meta`。
- 颜色、圆角、间距继续使用现有 CSS 变量，例如 `--space-*`、`--radius-*`、`--border-color`、`--secondary-bg`、`--tertiary-bg`、`--success-bg`、`--danger-bg`、`--info-bg`。

### UI 改动方式

run 明细进入面板时，优先采用“现有卡片内展开”或“同风格详情抽屉/弹窗”，不要做新的大屏、时间线、仪表盘或强烈不同的视觉组件。

建议第一版这样落地：

- 在任务卡片的 `runtime-panel` 内增加“查看最近回声”“查看验收详情”入口。
- 在“最近执行记录”的 `history-item` 右侧增加下拉/展开按钮，默认收起，展开后在同一条记录下方显示详情。
- 详情内容复用现有边框、背景、圆角和字体层级，只增加必要的信息块。
- 新增状态颜色只映射到现有语义色：完成用成功色，失败和验收未通过用危险色，等待和执行中用信息色，旧记录未验证用中性色。
- 移动端继续遵循现有 `@media (max-width: 900px)` 的纵向堆叠规则。

### 最近执行记录展开详情

每条历史记录保持当前紧凑摘要样式，右侧状态标签附近增加一个小型下拉按钮。点击后展开该条记录的详情区，不跳转页面。

展开区建议展示：

- 基础信息：runId、任务ID、目标 Agent、触发方式、开始时间、完成时间、耗时。
- 任务回声：显示 Agent 返回的 `【任务回声】` 原文或解析后的摘要。
- 系统验收：显示每条验收规则的通过/未通过状态和证据。
- 产物痕迹：显示日记、文件、话题或其他可追踪产物路径。
- 错误详情：仅在失败、等待超时、验收未通过时显示。

加载策略：

- 历史列表接口只返回摘要，保证页面刷新轻快。
- 展开某一条记录时，再通过 `GET /task-assistant/runs/:runId` 获取完整详情。
- 已展开过的详情可以在前端短暂缓存，避免重复点击反复请求。
- 如果是旧记录没有 runId，则展开区显示“旧记录未验证，缺少完整运行详情”。

### 不做的事

- 不新增与当前 AdminPanel 明显割裂的配色、阴影、字体、动画。
- 不把 run 详情做成独立风格的“审计系统”页面。
- 不把任务回声做成工程交付报告式表格；它应该像伙伴留下的可读回声，系统验收才使用结构化展示。
- 不在列表里直接塞入过长原始响应，长内容放进展开区或详情层。
