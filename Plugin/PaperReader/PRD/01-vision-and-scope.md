# 01. Vision And Scope

> 状态: Draft
> 所属: PaperReader Rust DDD PRD

---

## 1. 愿景重定义

PaperReader 新版本的目标不是“读 PDF”，而是：

**把多类型长文档转化为可解析、可追溯、可深读、可跨文档归纳的认知对象。**

系统定位：

- 面向学术论文、技术文档、网页正文、Markdown 文档、报告、规范文档等长文本材料
- 统一通过 MinerU 获取结构化解析结果
- 用 Rust 重建状态机、领域模型、工件系统与多文档能力

---

## 2. 核心问题

旧系统的关键问题不只是“PDF 太垃”，而是：

1. 输入被 `PDF` 概念绑死
2. 解析结果没有沉淀为足够强的统一中间表示
3. 单文档阅读、Query、骨架提取仍偏功能堆叠
4. 多文档能力几乎不存在
5. `paperId` 语义限制了系统未来扩展

因此，新系统要解决的不是单点 parser 问题，而是：

- 如何把多类型文档统一投影到一个稳定的认知模型
- 如何让单文档深读与多文档归纳共用同一套核心能力
- 如何让每个结论都能回溯到具体证据块

---

## 3. 产品目标

### 3.1 单文档目标

- 导入任意 MinerU 支持的长文档
- 自动生成结构骨架与阅读地图
- 支持目标驱动的深读、扫读、审核、总结
- 支持问答时返回证据来源

### 3.2 多文档目标

- 支持文档集合（Collection）管理
- 支持围绕同一问题对多文档做比较
- 支持跨文档综述与冲突检测
- 支持沿某个 claim 追踪支持与反驳证据

### 3.3 工程目标

- Rust 实现，强化类型边界与状态一致性
- 工件持久化明确、断点恢复明确
- 输入、结构、阅读、检索、归纳分层清晰

---

## 4. 范围（In Scope）

### 4.1 输入范围

当前版本支持的输入概念：

- 学术论文
- 技术报告
- 产品/接口规范
- Markdown 长文
- 网页正文导出内容
- 书籍章节/长篇文本材料
- MinerU 可正确处理的其他文档类型

### 4.2 核心能力范围

- 单文档导入与结构化
- 单文档阅读状态机
- 单文档证据检索
- 多文档集合管理
- 多文档比较与综述
- 冲突审计与证据对齐

### 4.3 工件范围

- 统一文档中间表示
- 结构树
- segment 清单
- 阅读状态
- 审核报告
- 集合级索引与归纳结果

---

## 5. 非目标（Out of Scope）

当前阶段明确不做：

- 本地备用解析器或解析 fallback
- 低保真纯文本模式兼容
- 完整 GraphRAG 图数据库方案
- UI 层重新设计
- 引用管理软件深度集成
- 自动训练/微调模型能力

---

## 6. 核心原则

1. **MinerU 单核原则**：解析统一由 MinerU 提供
2. **结构优先原则**：先得到统一文档表示，再进行阅读/检索/归纳
3. **目标驱动原则**：所有后续处理围绕 `goal` 进行
4. **证据优先原则**：所有结论必须能追溯到具体 segment/block/document
5. **单核多态原则**：单文档与多文档能力共用一个核心内核
6. **显式失败原则**：MinerU 不可用时任务失败并结构化上报，不伪装成功

---

## 7. 成功标准

### 7.1 单文档成功标准

- 能对一份长文档完成导入、结构化、深读与总结
- 最终报告可回溯到 segment 级证据
- 阅读状态支持中断恢复

### 7.2 多文档成功标准

- 能对一个文档集合生成集合概览
- 能围绕具体问题做跨文档比较
- 能输出冲突结论及对应证据来源

### 7.3 架构成功标准

- 新系统使用 `documentId/collectionId` 而非 `paperId`
- 中间表示与工件 schema 独立于输入文档类型
- API 与工件能同时支撑单文档和多文档流程
