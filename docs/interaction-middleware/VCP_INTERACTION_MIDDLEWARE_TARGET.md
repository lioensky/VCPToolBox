## 1. 文档目的

本文档不再从“某个平台接入怎么做”出发，而是从更高一层定义 VCP 的长期目标：

**VCP 不只是一个多平台接入中心，而是一个统一的交互中间层。**

它位于：

- 上游的人类用户、机器人平台、网页前端、外部系统
- 下游的模型、Agent、插件、工具、工作流

之间，负责完成统一的事件接入、状态管理、路由编排、消息投递和可观测控制。

本文档回答以下问题：

1. VCP 最理想的终局形态是什么。
2. 这个系统的核心对象应该如何定义。
3. 控制面和数据面应该如何分层。
4. 平台适配器在整个系统里应扮演什么角色。
5. 后续演进时，哪些是核心能力，哪些只是边缘协议细节。

---

## 2. 一句话定义

**VCP 是一个把多端交互、会话状态、Agent 路由、工具执行和消息投递解耦的交互中间层。**

它不是：

- 某个单独平台的机器人后台
- 某个模型的 HTTP 代理
- 一组零散 webhook 的集合

它应该是：

- 一个统一的事件入口层
- 一个统一的会话与身份映射层
- 一个统一的路由与执行编排层
- 一个统一的回复与投递层
- 一个统一的控制台与运维层

---

## 3. 核心设计原则

### 3.1 平台不是中心，交互流才是中心

系统不应围绕 QQ、钉钉、飞书、网页聊天分别设计主链路。

系统应围绕以下统一问题组织：

1. 进来的是什么事件
2. 事件属于哪个会话
3. 事件应该路由给谁
4. 需要哪些上下文、策略、工具和运行时
5. 产出的结果是什么标准回复
6. 回复应该如何投递、降级、重试和审计

只要核心链路里还充满 `if (platform === 'qq')` 这类判断，说明中间层仍未成熟。

### 3.2 新平台应该改边缘，不应该改核心

这是判断架构是否成型的最重要标准。

理想状态下，新增一个平台时，只需要新增或配置：

- 适配器 schema
- 入站编解码器
- 出站编解码器
- 能力声明
- 发送实现

而不需要修改：

- 会话系统
- 路由策略
- Reply 标准
- Outbox
- 审计与指标体系

### 3.3 控制面和数据面必须严格分离

控制面负责：

- 配置
- 密钥
- 能力声明
- 健康检查
- 绑定管理
- 审计与指标
- 重试与死信管理

数据面负责：

- 实时接入
- 事件归一化
- 会话解析
- 路由
- 运行时调用
- 回复投递

页面和管理逻辑不应侵入消息主链路。

### 3.4 平台适配器应尽量薄

平台适配器只应关心平台协议本身，不应持有业务逻辑。

适配器应该只负责：

- 鉴权
- 签名验证
- transport 连接
- 平台事件解码
- 平台回复编码
- 按 session 投递
- 平台能力声明

适配器不应该负责：

- Agent 选择
- 主题切换
- 业务绑定策略
- 审计记录
- 补偿重试逻辑

### 3.5 中间层必须优先标准对象，而不是平台私有对象

VCP 内部所有核心链路应只消费统一标准对象，不直接消费平台私有 payload。

上游应该统一成事件。
下游应该统一成回复和投递任务。

## 4. 目标能力边界

### 4.1 VCP 的职责

VCP 负责：

- 接入来自不同平台和终端的交互事件
- 将平台协议事件归一化成标准事件
- 管理会话、身份、绑定与上下文
- 按策略将事件路由给 Agent、插件、工具或工作流
- 将结果统一归一化为标准回复
- 按目标平台能力进行投递、降级、重试与审计

### 4.2 VCP 不负责的事情

VCP 不直接负责：

- 具体平台 SDK 的全部高级能力封装
- 某个终端专属的 UI 逻辑
- 大模型底层推理实现
- 单个平台完整业务后台
- 平台专属配置页面的大量业务堆砌

这意味着 VCP 应该保持“中间层”身份，而不是不断膨胀成每个平台的超级后端。

## 5. 核心对象模型

VCP 的理想形态应该建立在一套稳定的对象模型上，而不是围绕平台字段拼接。

### 5.1 Event

表示一切入站交互的统一载体。

一个 Event 至少包含：

- `eventId`
- `adapterId`
- `channel`
- `eventType`
- `occurredAt`
- `sender`
- `client`
- `session`
- `payload`
- `metadata`

无论来源是：

- QQ
- 钉钉
- 企业微信
- 飞书
- WebChat
- OpenAI 兼容 HTTP 调用
- 内部系统主动注入

都先转成统一 Event。

### 5.2 Session

Session 是中间层资产，不属于任何单一平台。

推荐双层设计：

- `externalSessionKey`
  - 平台原生会话键
  - 只描述平台事实
- `bindingKey`
  - VCP 的内部绑定键
  - 承载 Agent、Topic、用户映射等中间层语义

Session 至少应能承载：

- 当前绑定的 Agent
- 当前 Topic
- 当前身份映射
- 最近活跃时间
- 最近消息统计

### 5.3 Actor

Actor 表示系统中的交互参与者。

包括但不限于：

- 用户
- 群成员
- 机器人账号
- Agent
- 插件
- 工具执行器
- 人工坐席

系统不应只围绕“用户”和“机器人”两类角色设计。

### 5.4 Artifact

Artifact 表示一切可传递内容。

例如：

- 文本
- 图片
- 音频
- 视频
- 文件
- 卡片
- 表单
- 按钮
- 工具结果
- 结构化对象

Artifact 是 Reply 的构成材料，而不是平台附属字段。

### 5.5 Route

Route 用于描述一个事件应当被谁处理。

可能的目标包括：

- 某个 Agent
- 某个 Workflow
- 某个 Plugin
- 某个 Tool Pipeline
- 某个人工流程

Route 应由策略层决定，而不是由平台适配器决定。

### 5.6 Job

一切出站动作都应该变成 Job。

包括：

- 回复投递
- 主动通知
- 补发
- 重试
- 死信转移

Job 是实现可追踪、可审计、可重试系统的基础。

## 6. 目标分层架构

### 6.1 Ingress Layer

职责：

- 接收 webhook / ws / internal push / API injection
- 适配平台 transport
- 基础鉴权、签名验证、重放保护、限流

这一层只做入口控制，不做业务处理。

### 6.2 Adapter Layer

职责：

- 将平台原始消息解码为标准 Event
- 将标准 Reply 编码为平台原生消息
- 声明平台能力与限制
- 实现按 session 的投递动作

这是边缘协议层。

### 6.3 Normalization Layer

职责：

- 校验 schema
- 去重
- 内容标准化
- 提取 sender / client / session 关键信息
- 生成 traceId / requestId

这一层把平台协议完全转换为中间层对象。

### 6.4 Session / Identity Layer

职责：

- 解析或创建 session
- 管理 bindingKey 与 externalSessionKey 映射
- 管理用户、组织、平台身份映射
- 承载 topic、agent、上下文归属

这是系统中最容易失控、也是最关键的一层。

### 6.5 Routing / Policy Layer

职责：

- 根据 channel、session、sender、intent、rules 决定处理目标
- 选择 Agent / Workflow / Plugin / Tool
- 应用租户、白名单、权限、节流和策略

理想上，这一层不依赖平台私有字段。

### 6.6 Execution Layer

职责：

- 调用 ChatCompletionHandler、Agent、Plugin、Tool、Workflow
- 聚合执行结果
- 控制流式或非流式处理
- 处理错误、超时与中断

这里是“能力运行时”，不是“平台层”。

### 6.7 Reply Layer

职责：

- 统一生成标准 Reply
- 统一表示文本、多模态、操作指令、工具结果
- 附带 usage、topic、meta、toolEvents

回复一旦形成标准对象，后续平台只做编码与投递。

### 6.8 Delivery Layer

职责：

- 出站投递
- 能力降级
- 媒体上传
- 重试
- 死信
- 补偿

这一层决定系统是否具备生产级稳定性。

### 6.9 Control Plane

职责：

- 管理适配器、绑定、能力、密钥
- 查看最近事件、trace、审计记录
- 查看 outbox、死信、重试情况
- 做健康检查、回放、测试发送

这是给运营和开发看的“控制台”，不是简单配置页面。

### 6.10 Observability Layer

职责：

- 指标
- 审计
- Trace
- 错误聚合
- 链路可视化

应贯穿所有层，而不是附加在页面末端。

## 7. 控制面与数据面

### 7.1 数据面

数据面关注一条事件如何流转。

标准流如下：

1. 平台事件进入 Ingress
2. 适配器解码为标准 Event
3. 标准化、鉴权、去重
4. 解析 Session 与 Identity
5. 应用 Route 与 Policy
6. 执行 Agent / Plugin / Tool / Workflow
7. 生成标准 Reply
8. 写入 Delivery Job
9. 投递到目标平台
10. 记录审计与指标

### 7.2 控制面

控制面关注系统如何被管理。

应至少支持：

- 适配器注册、启停、测试、健康检查
- 会话绑定创建、修改、迁移、删除
- 身份映射维护
- 能力矩阵查看
- 密钥管理
- 审计查询
- Outbox 管理
- 死信清理
- Trace 回放

## 8. 平台适配器的理想契约

每个适配器应实现统一契约，而不是各写一套行为。

推荐最小契约如下：

### 8.1 元信息

- `id`
- `channel`
- `displayName`
- `transportMode`
- `capabilities`
- `sessionGrammar`

### 8.2 安全接口

- `authenticate(requestContext)`
- `verifySignature(requestContext)`

### 8.3 编解码接口

- `decodeInbound(rawPayload, context) -> Event`
- `encodeOutbound(reply, context) -> PlatformMessage[]`

### 8.4 投递接口

- `sendBySession(sessionDescriptor, platformMessages, context)`

### 8.5 诊断接口

- `healthCheck()`
- `getCapabilities()`

这样新增平台时，系统核心只依赖契约，不依赖平台细节。

---

## 9. 能力矩阵应成为一等资源

能力矩阵不应只是文档说明，而应成为运行时可读取的配置。

一个适配器至少应声明：

- 是否支持文本
- 是否支持图片
- 是否支持音频
- 是否支持视频
- 是否支持文件
- 是否支持卡片
- 是否支持按钮
- 是否支持引用回复
- 是否支持合并转发
- 是否支持流式消息
- 是否支持群聊
- 是否支持私聊
- 是否支持线程
- 是否支持主动推送

能力矩阵的直接用途：

- 出站降级
- 页面展示
- 文档提示
- 配置校验
- 自动测试

## 10. 理想的管理控制台

理想的前端不应只是“配置几个 JSON”。

应是一个面向流转和运维的控制台。

### 10.1 适配器视图

应支持：

- 向导式创建适配器
- 固定字段配置
- 密钥来源提示
- 一键测试连接
- 一键测试签名
- 查看能力矩阵
- 查看最近错误

### 10.2 会话视图

应支持：

- 查看活跃 session
- 查看 session 命中的 binding
- 查看绑定的 Agent / Topic / Identity
- 手动迁移绑定
- 手动修改 route

### 10.3 事件视图

应支持：

- 查看最近入站事件
- 看到事件是如何命中 route 的
- 看到使用了哪些工具和运行时
- 看到 reply 如何被标准化

### 10.4 投递视图

应支持：

- 查看 outbox
- 重试失败消息
- 查看死信
- 回放消息
- 查看平台降级结果

### 10.5 诊断视图

应支持：

- adapter 健康状态
- 每平台吞吐、成功率、延迟
- route 命中统计
- 运行时耗时
- 投递失败原因分布

---

## 11. 安全模型

作为中间层，VCP 必须统一管理安全能力。

至少应包含：

- adapter 级密钥
- 请求签名验证
- 时间戳容忍窗口
- nonce 重放保护
- IP 白名单
- 适配器级权限隔离
- 控制面与数据面权限分离

任何平台如果把这些能力分散到各自脚本里处理，长期都会不可控。

---

## 12. 演进路线

### 12.1 第一阶段：从“桥接”升级为“标准入口”

目标：

- 统一入站 Event
- 固化标准 Reply
- 固化 adapter 契约
- 把 `/internal/channel-ingest` 过渡为正式标准入口

### 12.2 第二阶段：会话与身份中间层化

目标：

- 固化 `externalSessionKey`
- 固化 `bindingKey`
- 独立 Identity Mapping
- Session / Topic / Agent 绑定稳定化

### 12.3 第三阶段：投递层生产化

目标：

- Outbox 完整
- 重试 / 死信 / 补偿可用
- 能力降级稳定
- 媒体上传与引用回复支持完善

### 12.4 第四阶段：控制台与可观测性完善

目标：

- 管理控制台从“配置页”进化为“交互控制台”
- trace、metrics、audit 联通
- 支持诊断、测试和回放

### 12.5 第五阶段：边缘平台快速接入

目标：

- 新平台只改适配器
- 核心链路保持稳定
- 形成可复制的平台接入模板

---

## 13. 非目标

为了防止架构继续膨胀，以下内容不应作为中间层核心目标：

- 为每个平台深度重做一套业务后台
- 把所有平台专有高级功能都抽象成统一对象
- 让控制台承担具体业务操作页面
- 把中间层做成另一个“大而全应用平台”

VCP 的核心任务仍然是：

**连接、编排、治理，而不是吞并所有业务。**

---

## 14. 最终判断标准

判断这套架构是否真正成型，可以看以下四个问题：

1. 新增一个平台时，是否只改边缘适配器。
2. 平台差异是否被限制在编解码和能力声明层。
3. 核心链路是否只消费标准对象。
4. 出问题时，是否能从事件、路由、回复、投递一路追到根因。

如果这四点能成立，VCP 才算真正成为“交互中间层”。

---

## 15. 最终愿景

最理想的 VCP，不是“支持很多平台的机器人后台”。

而是：

**一个让任何交互入口都能稳定接入、让任何能力执行都能统一编排、让任何回复都能被可控投递的中间层基础设施。**

换句话说：

**平台越来越多，核心越来越稳定。**
